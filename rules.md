# Протокол разработки проекта (Category-Operator-Determination)

## Основные принципы
1. **No Placeholders:** Никаких комментариев вида `// TODO`, `/* реализовать позже */` или недописанных функций. Весь код должен быть полностью рабочим и готовым к продакшену.
2. **Strict TypeScript & Python:** 
   - На фронтенде запрещено использование типа `any`. Все интерфейсы (interfaces) и пропсы должны быть строго типизированы.
   - На бэкенде обязательна типизация через Pydantic-схемы для всех входящих и исходящих API-запросов.
3. **Context Control:** Перед изменением любого файла перечитай его структуру. Не перезаписывай весь файл, если нужно изменить одну строчку. Декомпозируй длинные функции на чистые подфункции.
4. **UX/UI Excellence:** Верстка должна быть адаптивной (мобильные устройства + десктоп), аккуратной, в строгом современном b2b-стиле (Tailwind CSS, темные/светлые технологичные тона).

## Стек проекта
- **Frontend:** React, Vite, TypeScript, Tailwind CSS, Lucide-react (иконки).
- **Backend:** Python, FastAPI, Pydantic v2, Uvicorn.
- **AI Интеграции:** Gemini API (использование мультимодальных возможностей Google GenAI SDK для анализа изображений и генерации SEO-текста).

## Правила обработки ИИ-запросов (Gemini API)
- Всегда настраивай обработку ошибок (Error handling). Если API упало или ключ невалиден, интерфейс должен вывести понятную ошибку пользователю, а не падать в "белый экран".
- Результаты от Gemini (JSON с SEO-текстом и триггерами) должны валидироваться на стороне бэкенда через Pydantic до отправки на фронтенд.
